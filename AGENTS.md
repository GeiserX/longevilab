#### tRPC Procedures

All authenticated endpoints use `protectedProcedure` (validates session + user, throws `UNAUTHORIZED`). Public-facing endpoints use `publicProcedure`. All input validated with Zod schemas.

```typescript
// server/trpc/trpc.ts
export const createTRPCContext = async () => {
  const session = await auth.api.getSession({ headers: await headers() });
  return { db, session, user: session?.user ?? null };
};
```

#### File Uploads Bypass tRPC

tRPC does not support multipart form data. These two routes are the only non-tRPC API endpoints:

- `POST /api/upload` — Blood test PDF (max 20MB). Accepts optional `householdMemberId`.
- `POST /api/upload/health-export` — Apple Health ZIP (max 100MB, `nodejs` runtime). Accepts optional `allData=true` for full history.

Both authenticate via Better Auth session cookie, not tRPC context.

#### Naming Mismatch

Docker container names and DB name use the legacy `bloodwork` prefix (`bloodwork-app`, `bloodwork-worker`, `bloodwork-postgres`). The project itself is `longevilab`. Do not rename these — they match deployed infrastructure.

#### Household Member Scoping

Blood tests, health journal entries, hourly heart rate, and health data imports have an **optional** `householdMemberId` FK. Medications and supplements are user-scoped only (no household member). When `householdMemberId` is `undefined` in queries, all of the user's records are returned.

The selected member is stored in `sessionStorage` (key: `selectedMemberId`) via `SelectedMemberProvider` context.

## Best Practices

### Blood Test Processing Pipeline

Upload PDF → OCR → Regex Extraction → Unit Conversion → DB Insert. Three BullMQ queues process sequentially:

1. **`ocr-processing`** (concurrency 5): PDF → text via Tesseract (local, `eng+spa`) or Datalab.to (cloud). Lock 600s.
2. **`llm-extraction`** (concurrency 3): Text → biomarker array via deterministic regex with ~80+ Spanish/English aliases. `LLM_PROVIDER=none` is default — no LLM API calls. Lock uses BullMQ default.
3. **`health-export-processing`** (concurrency 1): ZIP → SAX streaming parse of `export.xml`/`exportación.xml` → daily aggregation → dedup → batch insert. Lock 1800s.

Retry: OCR/LLM get 3 attempts with exponential backoff (30s base). Health export gets 2 attempts with 60s fixed backoff.

### Apple Health Export Quirks

- Spanish iOS exports use `exportación.xml` with NFD-encoded Unicode in ZIP entry paths. The parser normalizes to NFC: `entry.path.normalize("NFC")`.
- SAX streaming avoids loading 100MB+ XML into memory.
- Records are aggregated per day (sleep stages summed, heart rate averaged, steps totaled).
- Deduplication: keeps latest per date/type, except blood pressure (keeps all readings).

### Unit Conversion

Programmatic conversion for 11 biomarkers with known factors (glucose, cholesterol panel, triglycerides: mg/dL ↔ mmol/L). Unknown units fall back to the extraction service. Canonical units are enforced per biomarker in the DB schema.

### Seed Data

`pnpm db:seed` populates ~80+ biomarkers, their reference ranges (by age/sex), and educational content. Uses `onConflictDoNothing()` — safe to run repeatedly.

| Variable | Required | Description |
|----------|----------|-------------|

| `HEALTH_EXPORT_WORKER_CONCURRENCY` | No | Health export queue concurrency (default: 1) |

Local Tesseract OCR is used by default. Set `DATALAB_API_KEY` for higher-quality cloud OCR.
